## Context

现有简历优化有4个独立tab：JD匹配、结构检查、深度润色、STAR改写。这些功能各自孤立，用户需要自己判断该用哪个。真实招聘场景中，简历经过一条筛选漏斗（ATS → HR → 业务主管 → 决策层），每一层的筛选标准完全不同。用户真正需要的是知道"我的简历会在哪一层被筛掉"。

后端已有成熟的 AI 调用链：`ResumeAiService` → `AiGateway` → LLM，以及 SSE 流式输出能力。现有 prompt 模板（`structure-analysis.md`）已经能做到段落级精准批注（引用原文 + 问题 + 建议 + 改写），新功能复用这个模式。

前端 `ResumeOptimizerView.vue` 已经是 tab 结构，新增一个 tab 成本低。`resumeStore` 使用 Composition API，扩展 state/actions 直接加即可。

## Goals / Non-Goals

**Goals:**
- 一次调用返回模拟4层漏斗的综合评估报告
- 覆盖现有结构检查未涉及的维度：量化密度、动词强度、成果导向比、红旗信号、职业叙事
- 逐条批注保持现有交互模式（引用原文 → 一键应用改写）
- 支持有JD和无JD两种模式（无JD时跳过岗位匹配维度，其余维度正常评估）
- 前端体检报告UI清晰展示漏斗各层评分 + 红旗/警告/亮点 + 逐条批注

**Non-Goals:**
- 不做ATS格式解析（检测PDF/DOCX格式兼容性），只分析文本内容中的关键词覆盖
- 不做简历导出/下载
- 不做历史版本管理
- 不做交互式引导（教练模式）
- 不修改现有JD匹配分析、深度润色、STAR改写的功能和接口

## Decisions

### 1. 单次AI调用 vs 多次调用

**选择：单次AI调用，一个prompt返回完整JSON**

理由：
- 现有结构检查已经是单次调用返回完整JSON，模式成熟
- 多次调用增加延迟，用户体验差
- LLM（GLM-4-flash）的输出长度足够支持完整报告
- 避免多次调用的编排复杂度

备选方案：拆成多次调用（ATS扫描、HR视角、技术视角、风险检测各调一次）， rejected 因为延迟翻倍且结果难以聚合。

### 2. 前端Tab方案

**选择：新增"简历体检"tab，放在现有3个tab之后**

理由：
- 保持现有tab不变，新功能独立添加，改动范围最小
- 用户可以先用体检快速定位问题，再切到深度润色逐条改写
- 未来如果体检体验好，可以考虑替换结构检查tab，但现阶段并行

备选方案：替换现有结构检查tab为一个综合tab，rejected 因为结构检查有独立使用场景，合并会丢失轻量入口。

### 3. 后端DTO结构

**选择：新建独立的 `HealthCheckupRequest` / `HealthCheckupResponse` DTO体系**

理由：
- 和现有 `StructureAnalysisResponse` 解耦，不破坏现有接口
- 体检报告的数据结构更复杂（多漏斗层、多维度、红旗/警告/亮点分类）
- 现有接口继续可用，不影响结构检查tab

### 4. Prompt设计

**选择：一个大prompt，输出结构化JSON**

Prompt结构：
1. 系统角色定义：同时扮演ATS系统、HR、技术面试官、风险评估师
2. 4个漏斗层各有独立的评估维度和评分标准
3. 逐条扫描每段经历，给出批注
4. 输出严格JSON schema

和现有 `structure-analysis.md` 的关系：体检prompt完全独立编写，不复用结构检查的5维度扫描，因为体检是从漏斗视角而非结构视角。

### 5. 无JD模式

**选择：JD可选，无JD时跳过岗位匹配相关维度**

无JD时：
- 不做ATS关键词匹配（没有JD就没有目标关键词）
- 不做岗位匹配度评分
- 其余维度（量化密度、动词强度、成果导向、红旗信号、职业叙事、信息架构）正常评估
- 综合评分只基于可用维度

## Risks / Trade-offs

**[单次调用输出过长]** → LLM可能截断或超时 → Mitigation：prompt中严格要求精简输出，每条批注限制在2-3句话；如仍超长，后续可拆分为多次调用

**[LLM评分不稳定]** → 同一简历多次体检分数可能不同 → Mitigation：评分区间给range而非精确值（如"65-75"），减少用户对精确分数的依赖；维度评分比综合分数更有参考价值

**[体检与结构检查功能重叠]** → 用户困惑该用哪个 → Mitigation：UI上明确区分定位——体检是"快速全面诊断"，结构检查是"深度结构修复"；长期可考虑合并

**[无JD模式下ATS维度缺失]** → 体检报告不完整 → Mitigation：无JD时明确标注"未提供JD，ATS关键词匹配维度已跳过"，并建议用户提供JD获得更完整评估
